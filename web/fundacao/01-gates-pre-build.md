# Gates Pre-Build do Web

## 1. Objetivo

Formalizar o que precisa estar suficientemente definido antes de começar a construir o produto web de forma séria.

## 2. Decisão principal

Faz sentido `parar antes do build` para fechar as decisões estruturais.

Motivo:

- evita retrabalho caro
- evita inconsistência entre frontend, backend e integrações
- reduz risco de refazer UI inteira depois
- melhora a qualidade do beta com profissionais reais

## 3. Gates obrigatórios

### Gate 1: Stack end-to-end

Precisa fechar:

- frontend
- backend
- banco
- fila
- storage
- auth
- observabilidade
- testes

### Gate 2: Estratégia de integração

Precisa fechar:

- vídeo
- transcript
- LLM
- assinatura eletrônica
- pagamento

### Gate 3: Branding e direção visual

Precisa fechar:

- nome visual temporário ou definitivo
- paleta principal
- tipografia
- linguagem visual do produto
- tom da marca

### Gate 4: Layout e composição

Precisa fechar:

- grids
- densidade das telas
- padrões de lista
- padrões de detalhe
- padrões de formulário

### Gate 5: Sinais de mercado

Precisa fechar:

- quais padrões parecem funcionar
- o que deve ser evitado
- o que precisa virar diferencial real

## 4. O que não precisa travar 100%

- naming definitivo da marca
- identidade visual completa de marketing
- motion avançado
- páginas mobile nativas
- enterprise features

## 5. Critério de saída deste estágio

O build principal do web pode começar quando houver:

- stack recomendada aprovada
- shortlist de integrações aprovada
- direção visual aprovada
- modelo de layout aprovado
- benchmarks de mercado traduzidos em decisão prática
